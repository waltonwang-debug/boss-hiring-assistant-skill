# 站内沟通服务

## 目标

只负责 BOSS 站内聊天推进，例如：

- 索要附件简历
- 确认候选人意向
- 收集可约时间
- 发送标准 FAQ 回复

不负责：

- 全量筛选
- 自动创建飞书日程

## 进入条件

只有在下面任一条件满足时才进入：

1. 用户明确要求“继续沟通这些候选人”
2. 每小时汇报中，用户明确选择了可推进沟通的候选人

进入本服务前，应当先告诉用户：

- 这是三类服务中的第二类
- 它只负责站内沟通
- 不会自动进入飞书约面，除非用户后续明确要求

## 消息发送规则

所有 BOSS 站内消息发送前，必须先读取：

- [boss-send-recipe.md](./boss-send-recipe.md)

并严格按其中的唯一主路径执行。

## 强约束动作

以下动作都属于强约束动作：

- 打开候选人详情
- 切换聊天线程
- 打开聊天线程
- 发送标准消息
- 索要附件简历
- 收集候选人可约时间
- 确认面试时间
- 发送飞书链接
- 接收候选人发送的附件简历

这些动作必须：

1. 先准备最终 payload
2. 使用 `web-access` 主线程快路径
3. 先确认候选人线程已经成功打开
4. 做一次短时成功确认
5. 失败就进入可恢复暂停态

不允许：

- 长时间探索
- 连续试很多 DOM 变体
- 起子代理研究前端框架

如果“打开候选人线程”失败，正确做法应是：

1. 先走标准自动打开路径
2. 再走有限备用路径
3. 若仍失败，先走低 token 的分层自动诊断
4. 只有诊断后仍不能定位，才暂停并请用户确认页面状态

不要一失败就把点击候选人这件事转交给用户。
不要在诊断阶段做大段复杂 eval 和多轮长时间试错。

## 默认消息策略

默认优先使用固定模板。
只有低风险 FAQ 且没有固定模板时，才允许受控生成简短回复。

## 附件简历闭环规则

当龙虾向候选人发送“请发送附件简历”的消息后，流程不能停留在“已请求”状态，必须继续跟进到闭环。

闭环顺序：

1. 记录该候选人状态为 `attachment_requested`
2. 后续扫描时，优先检查该候选人线程是否有新的附件简历消息
3. 如果候选人已发送附件简历：
   - 更新状态为 `attachment_sent_by_candidate`
   - 检查聊天界面中是否出现“同意/接收”按钮
4. 若存在“同意/接收”按钮：
   - 点击该按钮完成接收
   - 更新状态为 `attachment_received`
5. 接收完成后，再把候选人推进到下一步：
   - 人工复核
   - 继续沟通
   - 或准备约面

## 附件简历跟进优先级

对已经处于下面状态的候选人，后续扫描时应提高优先级：

- `attachment_requested`
- `attachment_sent_by_candidate`
- `attachment_receive_pending`

也就是说，龙虾不仅要发现新未读候选人，还要跟进这些“已请求简历但尚未收完”的候选人。

## 附件接收的暂停条件

如果候选人已经发送附件简历，但“同意/接收”按钮点击失败：

- 不要忽略该附件
- 不要直接把候选人当成“未发送简历”
- 应进入 `paused_for_attachment_acceptance`

暂停时必须告诉用户：

- 候选人已发送附件简历
- 当前卡在“同意/接收”动作
- 简历尚未完成接收
