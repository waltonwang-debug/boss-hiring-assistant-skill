# 首次接入流程

## 目标

让用户只用一句自然语言请求，就能把这个 skill 接入龙虾，并进入正确的服务模式。

## 首次接入顺序

1. 先向用户介绍三类独立服务
2. 推荐用户先从 `boss-screening` 开始
3. 询问或判断用户本轮要启动哪一种服务
4. 按该服务引导对应的强制前置条件
5. 前置条件完成后，再开始自动读取当前账号下的在线岗位和 JD

前面任一步没有完成，都不要提前进入后面的步骤。

## policy 文件缺失时的处理

如果当前环境里没有现成的：

- `company-policy.yaml`
- `approval-policy.yaml`

必须先：

1. 自动读取当前在线岗位和 JD
2. 基于 JD 生成一版默认 policy
3. 再把这版默认 policy 给用户确认

不要在这一步直接问用户“筛选标准是什么”。
用户在这一步更适合补充的是：

- 公司特有的人才观
- 必须卡掉的硬条件
- 需要人工审批的边界条件
- 哪些情况必须人工复核

## 首次接入时必须说明的三类服务

1. `boss-screening`
用于简历筛选、分类和目标进度汇报。
推荐首次先从这里开始。
需要顺带告诉用户：

- 可筛选的范围包括主动进入聊天列表的未读候选人、推荐页面中的本轮未处理候选人、以及具备搜索权益时的搜索结果
- 核心规则是“每个 JD 每天要筛选出多少个符合要求的候选人”
- 时间窗口和汇报频率只是辅助设置
- 用户可以覆盖默认时间节奏
- 如果本地没有现成 policy，会先读取在线 JD 并自动生成默认筛选标准，再请用户补充差异

2. `boss-chat`
用于 BOSS 站内沟通，例如索要附件简历、确认意向、收集时间。
只有在用户明确要求沟通候选人时才启动。

3. `boss-scheduling`
用于候选人已确认时间后的飞书约面。
只有在用户明确要求约面时才启动。

## 服务选择后的前置条件引导

### 如果用户选择 `boss-screening`

只引导这些前置条件：

1. 由龙虾代为禁用或移除 `browser-use` 等浏览器 skill
2. 由龙虾代为安装 `web-access`
   - GitHub 链接：`https://github.com/eze-is/web-access`
3. 切换到 `web-access`
4. 验证 `web-access` 已接管
5. 确认使用 Chrome / Chromium
6. 确认用户已登录 BOSS 网页端

不要提前引导飞书 bot。

在这一步不要默认新开 Boss 页面。
应先让龙虾在当前浏览器会话中找到已登录的 Boss tab，并复用该 tab。

对于第 1 步，不要先问用户“是否已禁用 browser-use”。
应当先执行禁用，再把结果汇报给用户。

同时必须向用户说明：

1. 本服务默认可筛选三类来源：
   - 主动聊天/主动投递中的未读候选人
   - 推荐页面中的本轮未处理候选人
   - 搜索页面中的本轮未处理候选人（若账号有搜索权益）
2. 本服务需要先确认每个 JD 的每日目标候选人数
3. 默认辅助节奏是：
   - 周一到周五
   - 09:00-17:00
   - 每 60 分钟汇报一次
4. 用户若有自己的执行窗口或汇报节奏，可以立即覆盖默认值

如果每日目标人数还没有确认，不要开始筛选。
如果用户问“会不会重复扫旧候选人”，要明确回答：默认不会，除非候选人有新的未读消息、上一轮未补齐、或用户点名要求重扫。

### 如果用户选择 `boss-chat`

引导：

1. 完成 `boss-screening` 同样的通用前置条件
2. 再确认本轮要推进沟通的候选人范围

不要提前引导飞书 bot。
如果本轮要沟通的候选人范围未明确，不要开始发消息。

### 如果用户选择 `boss-scheduling`

引导：

1. 完成通用前置条件
2. 确认本轮要约面的候选人
3. 只有在第一次真的要创建飞书日程时，再引导飞书 bot 配置与授权

如果候选人范围未明确，或候选人尚未确认时间，不要开始约面流程。

## 安装链接规则

默认由龙虾代为安装 `web-access`。
只有在龙虾代装失败时，才回退为让用户手动安装。

当需要用户手动安装 `web-access` 时，不要让龙虾自己去搜索。
直接向用户提供固定链接：

- `https://github.com/eze-is/web-access`

## 不要在 onboarding 里提前做的事

不要在 onboarding 阶段提前：

- 读取中文话术模板
- 追问飞书 bot 细节
- 追问默认参会人
- 直接进入发消息
- 直接进入飞书建会

## onboarding 阶段只需要最少确认

如果自动发现岗位成功，只需要确认：

- 用户想先用哪一种服务
  - `boss-screening`
  - `boss-chat`
  - `boss-scheduling`

如果用户没有指定，默认进入 `boss-screening`。

## 默认值

如果用户没说自己的辅助节奏，自动采用：

- 周一到周五
- 09:00-17:00
- 每 60 分钟汇报

这些默认值都允许用户覆盖。
但在进入 `boss-screening` 前，仍然要先明确每个 JD 的每日目标候选人数。

## 飞书相关延后

只有在第一次真的要创建飞书日程时，才去确认：

- 是否已配置 bot
- 是否已完成 bot 聊天内授权
- 默认参会人设置
